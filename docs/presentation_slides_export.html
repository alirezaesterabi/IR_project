QMUL IR Project 1

1. Problem Motivation

  • Sanctions search is a high-stakes IR problem: missing a relevant entity can have legal and regulatory consequences.
  • Users do not always search in the same way. Some queries are exact lookups, while others are descriptive or investigative.
  • That means a single retrieval strategy is unlikely to be enough across the full task.
  • Our goal was to build a system that can support multiple query behaviours rather than a simple keyword search demo.

Main challenge

We are not searching tidy documents. We are searching entities with aliases, identifiers, nested metadata, and uneven descriptions gathered from many sources.

Presentation angle: start by framing this as a real compliance retrieval problem, not just a classroom ranking exercise.

Why this project matters 2

2. Why OpenSanctions Is Challenging

  • The dataset is large, heterogeneous, and built from many international sources.
  • Records are nested and multi-field rather than flat, so they are not directly ready for IR indexing.
  • The same entity may appear with different aliases, identifiers, descriptions, or source-specific metadata.
  • Some entity types are easy to identify through strong identifiers, while others depend on names or descriptive text.

Why this matters for retrieval

  • Exact matching is necessary for identifier lookups.
  • Lexical ranking is useful for names and aliases.
  • Semantic retrieval is needed when wording does not overlap.

3. Preprocessing Became the Foundation

  • We built a streaming parser so the raw file could be processed safely without loading everything into memory.
  • Each nested record was flattened into a retrieval-ready document representation.
  • We preserved a clear distinction between searchable text, raw identifiers, and structured metadata.
  • This preprocessing stage became the base that every later component depends on.

Processed document fields

  • `doc_id`
  • `caption`
  • `schema`
  • `text_blob` for lexical retrieval
  • `tokens` for BM25
  • `identifiers` for exact matching
  • `metadata` for filters and context
  • `embedding_text` for dense retrieval
Key design choice: we did not index raw JSON directly. We created an intermediate document format shaped around retrieval needs.
Representation before ranking 4

4. Text Processing Method

Field extraction stage

  • Group `name`, `alias`, and `previousName` as identity-bearing text.
  • Collect keyword-style fields such as sector, topics, legal form, and position.
  • Flatten free-text fields like `notes`, `description`, and `summary`.
  • Traverse nested sanctions and address objects to recover authority, reason, city, and address content.

Token-processing stage

  • Unicode normalisation, lowercasing, punctuation cleanup, and whitespace normalisation.
  • Light normalisation for names so identity signals are preserved.
  • Stop-word filtering and spaCy lemmatisation for free text to improve recall.
  • Identifiers kept raw; metadata kept separate from the main lexical text.
The final `text_blob` is assembled in a meaningful order: names first, then keyword fields, then descriptive text, then sanctions and address content. This gives BM25 and TF-IDF a cleaner lexical view of each entity.

5. Query Types and Examples I

Type 1: Exact Identifier

Query by exact field value such as IMO, MMSI, or registration number.

Example: `IMO9296822`

Metric: `Precision@1`, `MRR` | Secondary: `MAP`

Type 2: Name / Alias

Lookup by entity name, alias, or historical name with possible variation.

Example: `Angelica Schulte`

Metric: `Precision@1`, `MRR` | Secondary: `MAP`

Type 3: Semantic / Descriptive

User describes the target without knowing the exact entity name.

Example: `Russian crude oil tanker evading sanctions in the Baltic Sea`

Metric: `nDCG@10` | Secondary: `MAP`, `Recall@10`

Type 4: Relational / Graph

Question about links between entities, ownership chains, or network relations.

Example: `What vessels is Ketee Co Ltd linked to?`

Metric: `nDCG@10` | Secondary: `MAP`, `Recall@10`

Different search behaviours, different retrieval needs 6

6. Query Types and Examples II

Type 5: Cross-Dataset Deduplication

Find records of the same entity across different sources and list entries.

Example: `SAGITTA sanctions all sources`

Metric: `Recall@10`, `Recall@20` | Secondary: `MAP`

Type 6: Jurisdiction / Filter

Search constrained by authority, programme ID, country, or other metadata.

Example: `OFAC sanctioned vessels Russian oil 2025`

Metric: `Recall@10`, `Recall@20` | Secondary: `MAP`

Type 7: RAG Summarisation

Return a grounded narrative answer built from retrieved evidence, not only a ranked list.

Example: `Summarise all sanctions on the SAGITTA vessel`

Metric: `RAGAS faithfulness` | Secondary: `answer relevance`

Overall metric logic

Different query types need different metrics because they represent different information needs. At the system level, `Recall@10` is still an important overall measure because missing a relevant entity is costly in the sanctions domain.

7. Two Retrieval Lanes

Architecture flow adapted from the assignment system diagram
Stage 1: Preprocessed corpus Each entity is represented through lexical text, identifiers, metadata, and `embedding_text`.
→
Stage 2: Classical IR lane Identifier lookup, BM25, and TF-IDF over the lexical representation.
Stage 3: Dense retrieval lane Sentence embeddings and ChromaDB for semantic nearest-neighbour search.
→
Stage 4: Hybrid ranking The two ranked outputs will later be combined rather than forcing one lane to dominate.
This slide shows why the architecture needs two retrieval lanes before fusion.

Classical IR lane

  • Uses exact identifiers, BM25, and TF-IDF.
  • Best for strong lexical evidence such as names and identifiers.
  • Interpretable and useful as the baseline retrieval stage.

Dense retrieval lane

  • Uses `embedding_text`, sentence-transformer embeddings, and ChromaDB.
  • Helps when relevant records do not share the same vocabulary as the query.
  • Adds semantic evidence for harder descriptive and relational queries.

8. RRF Hybrid Ranking

Architecture flow adapted from the assignment system diagram
Classical output Ranked list from Identifier, BM25, or TF-IDF.
Dense output Ranked list from semantic nearest-neighbour search.
→
Stage 4: Reciprocal Rank Fusion Combine rank positions rather than trying to merge incompatible raw scores.
→
Final hybrid results One fused ranking used for evaluation and later extensions.
This is the step that turns two separate retrieval outputs into one hybrid result list.

Why RRF is a good choice

  • Lexical and dense systems produce different score scales.
  • RRF combines ranks, so it avoids fragile score normalisation.
  • It is a practical way to capture complementary strengths.

What this means conceptually

  • We do not force one model family to solve every query type.
  • The hybrid stage brings exact, lexical, and semantic evidence together.
  • This is the bridge from separate systems to one retrieval pipeline.

9. Building the Ground Truth

Automatic or semi-automatic qrels

  • Type 1: exact identifier match
  • Type 2: name / alias / previous-name matching
  • Type 5: same entity across datasets
  • Type 6: metadata filters such as programme, country, or schema

Pooling for types 3 and 4

  • Type 3: descriptive queries cannot be judged by one field match
  • Type 4: relational queries depend on links and context
  • Run multiple systems independently
  • Take top results from each system
  • Merge and deduplicate them into one judging pool
  • Assign manual relevance labels to the pooled set
Pooling is how we make manual judgement practical: instead of reading the whole collection, we judge a smaller candidate set proposed by several retrieval systems.

Type 7 is different again: it is not standard qrels-based evaluation because the output is a grounded narrative rather than only a ranked list.

11. Takeaways and Next Steps

  • We turned a complex real-world dataset into a retrieval-ready corpus through structured preprocessing.
  • We designed the system around seven query behaviours instead of assuming one search pattern.
  • We implemented complementary retrieval lanes: exact, lexical, and dense.
